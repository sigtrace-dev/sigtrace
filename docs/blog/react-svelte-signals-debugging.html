Reactivity is Magic (Until You Have to Debug It): Visualizing React & Svelte 5 Signals

By the SigTrace Contributors

Fine-grained reactivity is taking over front-end development. Svelte 5 has retired its compiler-based dirty-checking in favor of Runes, and React developers are increasingly turning to Preact Signals to bypass re-renders entirely. The speed is undeniable: your UI updates only when and where the underlying state changes.

But when state updates cascade into infinite loops or triggers refuse to fire, reactivity quickly turns from magic to a debugging nightmare. Standard debugger stacks are filled with framework internal engine frames, masking the code that actually scheduled the update. That's why we are excited to announce official React and Svelte 5 support for SigTrace.

The Problem: The Reactive Black Box

Consider a typical reactive graph. A user action triggers a state change (a signal write). That write updates two derived values, which in turn trigger three separate rendering effects. If one of those effects inadvertently updates another signal, you get an infinite cycle.

In standard browser devtools, you see the crash or the frozen page, but you cannot easily trace the causal chain back to the root trigger. You are left placing dozens of console.log() statements or stepping blindly through minified library code.

The SigTrace Approach: Rather than viewing state as a static snapshot, SigTrace instruments your signal creators at build time. It logs the exact trigger path and matches every reactive transaction back to the line of code that caused it.

How It Works in React (Preact Signals)

Preact Signals allow React developers to write highly performant applications by updating DOM nodes directly, bypassing React's virtual DOM diffing. When you register the SigTrace Vite plugin, your Preact signals are monitored in real-time:

// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import { sigTrace } from '@sigtrace/vite';

export default defineConfig({
  plugins: [react(), sigTrace()]
});

Once registered, any signal(), computed(), or effect() declared in your component files is automatically instrumented. You can see the name of the signal, its component context, and the full causal chain of updates directly in your VS Code sidebar.

How It Works in Svelte 5 (Runes)

Svelte 5's Runes syntax compiles directly into native signal structures. By adding SigTrace to your Vite config, Svelte's $state, $derived, and $effect are automatically mapped to the SigTrace visualization panel:

// Svelte component declaration
let count = $state(0);
let doubled = $derived(count * 2);

$effect(() => {
  console.log(`Doubled count is: ${doubled}`);
});

SigTrace captures the transition of the Svelte signal graph. When Svelte schedules an effect re-evaluation, SigTrace shows the exact sequence: count (write) ➔ doubled (re-evaluate) ➔ effect (run). Clicking on any step in the VS Code panel jumps your editor cursor directly to the corresponding rune declaration.

Getting Started in 2 Minutes

To try the new React and Svelte 5 integrations, simply add the dev dependencies and wrap your vite config:

# Install core tracer & plugin
npm install --save-dev @sigtrace/core @sigtrace/vite

# Start your server normally
npm run dev

Make sure you have the SigTrace VS Code Extension installed from the Marketplace. As soon as your browser loads the app, the VS Code sidebar will light up with the live reactivity log.

Reactivity doesn't have to be a black box. Give SigTrace a try, and see what your signals are doing under the hood.
